# runsc-prelaunch

A dstack prelaunch script that installs gVisor's `runsc` and registers it as a
Docker runtime, without modifying the dstack base image. Validates the
"gVisor is provisionable on stock dstack today" claim from
[isolation-probe](../../isolation-probe.md).

## Status

Verified working on `dstack-dev-0.5.9` / `DStack 0.5.9 (scarthgap)`, kernel
`6.9.0-dstack`. Test CVM provisioned with this script; `runsc release-20260420.0`
installed at `/dstack/persistent/bin/runsc`; `docker info` registers it
alongside `runc` and `sysbox-runc`; a container started with
`docker run --runtime=runsc alpine uname -a` reports gVisor's synthesised
kernel string `Linux ... 4.4.0 #1 SMP Sun Jan 10 15:06:54 PST 2016` instead
of the host's `6.9.0-dstack` — the irrefutable signature of Sentry
mediating syscalls.

## Why this works for attestation

- **The script's hash** is part of the measured launch payload — dstack
  records what was passed via `--pre-launch-script`.
- **The runsc binary** is pinned by sha512 inside the script, so the bytes
  that land in `/usr/local/bin/runsc` are fixed. A `latest` URL that ever
  served different bytes would fail verification and the script would exit.
- **Docker daemon.json** is generated by the script itself, not modified
  externally.

A relying party walking the trust chain compares the script's recorded hash
to the public copy in this repo, sees the pinned sha512, and concludes the
runsc binary running on the host is the published one — no runtime trust in
network-fetched bytes beyond the verification step.

## Deploy

```bash
phala deploy \
  --name <name> \
  --compose your-compose.yml \
  --pre-launch-script apps/runsc-prelaunch/prelaunch.sh
```

After provisioning, SSH in (via the dstack TLS gateway sidecar) and verify:

```bash
which runsc
runsc --version
docker info | grep -iE "runtimes|default runtime"
docker run --rm --runtime=runsc hello-world
```

## Use with tee-daemon

Set the daemon's preferred runtime to `runsc` so tenants come up under it:

```yaml
services:
  tee-daemon:
    image: ghcr.io/amiller/tee-socket-proxy:latest
    environment:
      - DAEMON_CONTAINER_RUNTIME=runsc
      # ... other config
```

`/_api/substrate` will then report `"effective_runtime": "runsc"`. Tenants run
under gVisor; the [isolation-probe](https://github.com/amiller/dstack-webhost/tree/main/apps/isolation-probe)
will show a different signature in `/proc/self/uid_map` and `user_ns` than
sysbox-runc — gVisor synthesises those from its own Sentry process.

## Caveats

- **Performance.** The ptrace/systrap platform is slower than KVM, which
  isn't available inside dstack TDX (no nested KVM). Syscall-heavy workloads
  will feel it.
- **Compatibility.** gVisor implements most syscalls in userspace; some are
  partially or unimplemented. Test the workloads you actually run.
- **Pin maintenance.** Bump `RUNSC_SHA512` in lockstep with the URL. The
  `latest` URL will eventually move; pin a versioned URL like
  `releases/release/<date>.0/x86_64/runsc` for production-grade reproducibility.
