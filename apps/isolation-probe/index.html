Isolation probe

This page is served by an ordinary tenant of dstack-webhost. Its job is to expose kernel-namespace evidence about itself so a relying party can corroborate — independently — what the substrate claims about which OCI runtime is mediating tenant syscalls inside the CVM.

The substrate is the small daemon that brings up tenant containers. It claims a runtime via /_api/substrate. This page fetches that claim and the tenant's own /proc, then checks the two against each other.

How to read this

  • uid_map — under sysbox-runc, root inside the container (UID 0) is mapped to a non-root UID on the host (e.g. 0 296608 65536 = container UIDs 0–65535 → host UIDs 296608+). Plain runc shows 0 0 4294967295 — root inside is root outside. The shifted form is the signature of an active user-namespace remap.
  • user_ns / pid_ns / mount_ns — namespace identities. Distinct values from the host = real namespace boundaries.
  • effective_runtime on the substrate — the daemon's stated choice of OCI runtime for tenants. This page corroborates that choice from inside one tenant; a malicious substrate could lie in /_api/substrate but cannot fake the tenant's own kernel-namespace view.

Why this matters

A multi-tenant TEE makes a strong claim — "your code ran here, others' code can't tamper with yours" — but that claim is only as good as the isolation between co-tenants. The TEE attestation says what code is running; it does not by itself say what's keeping co-tenants from each other's secrets and files. That's an OCI-runtime job.

Two relevant runtimes: sysbox-runc hardens the existing kernel boundary (user-namespace remap, virtualised /proc and /sys, scoped capabilities). gVisor goes further: a userspace kernel mediates syscalls so most kernel CVEs are unreachable. gVisor is stronger, but on Phala dstack TDX CVMs nested KVM isn't exposed, so gVisor would have to ship in the dstack base image (a coordinated change). sysbox-runc is registered as a Docker runtime on stock dstack today, so it's the isolation layer that's deployable now without changing the base image.
